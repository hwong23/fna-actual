href="https://www.opengroup.org/archimate-forum/archimate-overview " target="_blank" rel="nofollow noopener">ArchiMate</a>® notation and goes on to use a ‘driving view’ to generate output from the <a href="https://www.archimatetool.com" target="_blank" rel="nofollow noopener">Archi</a> enterprise architecture modelling tool. This can be used to repeatedly create consistent looking architecture deliverables on a project. This example uses Archi and its <a href="https://github.com/archimatetool/archi-scripting-plugin/wiki" target="_blank" rel="nofollow noopener">jArchi</a> scripting language,&nbsp;although other EA tools could achieve something similar.&nbsp;</p><p>It was created by the need to offer a trusted, up to date, document that can easily be accessed by non-tool users. The use of Markdown as the output format was the simplest one to try first, although this can easily be converted to MS Word / PDF etc.</p><h3>Defining a document</h3><p>An architecture deliverable document is often a structured set of views and accompanying descriptions. This is modelled using a simple ArchiMate view that uses a set of concepts to represent each section or subsection, and a means of representing the order. Archi supports creating ‘view of views’ by just dragging a view in from the model tree onto a view (as a ‘View Reference’).</p><p>I chose to use the grouping concept as these allowed for triggering relationships between them to show the order of sections in the document. Each View Reference needs to reside in a group, even if there’s only one View Reference.</p><h3>Generating the output</h3><p>Archi has its own reporting capability using Jasper Reports or generating to HTML, but I found this to be quite complex to generate specific documentation. Archi supports scripting using its Patreon supported jArchi javascript based utility.</p><p>Here’s a simple driving view to explain the components:</p><div class="slate-resizable-image-embed slate-image-embed__resize-full-width"><img alt="No alt text provided for this image" data-media-urn="" data-li-src="https://media.licdn.com/dms/image/C4E12AQFObQquHCH0Pw/article-inline_image-shrink_1500_2232/0/1570723503325?e=1680134400&amp;v=beta&amp;t=-zyp4iwdwrHoB7D79_PV7Q5BqCijYS7ZxOnfE0y40g4" src="//:0"></div><p>The script uses such a driving view and starts with the group that has no incoming trigger (Section One, in this example). It follows the triggers until it reaches a group with no outgoing trigger (Section Two). If there are sub groups (Transition and Target), it will also create a sub levels (level 2, here) for these.&nbsp;</p><p>Any documentation entered in the properties for the groupings will be extracted (representing its section text). The script generates an image for each Reference View it finds inside the groupings.</p><p>Here's an example of the document rendered by MacDown:</p><div class="slate-resizable-image-embed slate-image-embed__resize-full-width"><img alt="No alt text provided for this image" data-media-urn="" data-li-src="https://media.licdn.com/dms/image/C4E12AQGt_EnlAD9zJQ/article-inline_image-shrink_1500_2232/0/1570723667456?e=1680134400&amp;v=beta&amp;t=h35IR1h6lLkbFIVpIVC4YFDSd6-EEhTTGt6hWM9eFlQ" src="//:0"></div><h3>Conclusions</h3><p>It is possible to make a driving view conform to an organisation’s standard for documentation structure. Each project or initiative would then only need to have the appropriate views created and dragged in. This is a great way to drive conformance and to do completion checks for a design review. This could be scripted or just done manually.</p><p>There would have to be a balance on how much design is done in ArchiMate and Archi compared to adding text later directly into the document using a more natural word processor, but that’s always the challenge. A recent engagement established the guidance that it was best to only have true architecture content in the model, and project / contract related text added directly into the document afterwards (or as part of a template).</p><p>Using the script to repeatedly create initial versions of a document until all the views are in place is useful, even if manual fixing and layout is done at the end.&nbsp;Generating a perfect document is rarely possible, or worth the effort of making it happen.</p><h3>Further Information</h3><p>There is more detail on my blog including the scripts and an
